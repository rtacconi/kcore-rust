# Security Model

This document covers the security architecture, threat mitigations, and developer workflow for the kcore-rust codebase.

## PKI and Certificate Identity

All inter-component communication uses mutual TLS (mTLS). Certificates are generated by `kctl create cluster` and signed by a self-managed CA.

### Certificate identities

| Certificate       | Common Name (CN)          | Key Usages          | Validity |
|--------------------|---------------------------|----------------------|----------|
| CA                 | `kcore-cluster-ca`        | CA                   | 10 years |
| Controller         | `kcore-controller`        | ServerAuth, ClientAuth | 1 year   |
| kctl (CLI)         | `kcore-kctl`              | ClientAuth           | 1 year   |
| Node               | `kcore-node-<host>`       | ServerAuth, ClientAuth | 1 year   |

Validity periods are defined in `crates/kctl/src/pki.rs` (`CA_VALIDITY_DAYS`, `CERT_VALIDITY_DAYS`).

### Private key distribution

Only the minimum required keys are sent to each node during `kctl node install`:

- **All nodes** receive: `ca.crt`, `node.crt`, `node.key`
- **Controller nodes** additionally receive: `controller.crt`, `controller.key`
- **Never sent** to any node: `ca.key` (stays on the operator machine), `kctl.crt`/`kctl.key`

### File permissions

Private key files (`.key`) are written with mode `0o600` (owner-only). Certificates (`.crt`) use `0o644`.

## TLS Enforcement

Both `kcore-controller` and `kcore-node-agent` refuse to start without TLS configuration unless the operator explicitly passes `--allow-insecure`. When running in insecure mode, a prominent `WARN` log is emitted at startup.

```yaml
# Required in controller.yaml / node-agent.yaml for production:
tls:
  caFile: /etc/kcore/certs/ca.crt
  certFile: /etc/kcore/certs/<service>.crt
  keyFile: /etc/kcore/certs/<service>.key
```

## gRPC Authorization (CN-based)

Every gRPC method checks the caller's certificate Common Name before processing the request. Authorization is enforced per-method in the `auth` module (`crates/controller/src/auth.rs`, `crates/node-agent/src/auth.rs`).

### Controller RPC authorization

| RPC Method            | Allowed Callers           |
|------------------------|---------------------------|
| `RegisterNode`         | Nodes (`kcore-node-*`)    |
| `Heartbeat`            | Nodes (`kcore-node-*`)    |
| `SyncVmState`          | Nodes (`kcore-node-*`)    |
| `CreateVm`             | kctl (`kcore-kctl`)       |
| `DeleteVm`             | kctl (`kcore-kctl`)       |
| `SetVmDesiredState`    | kctl (`kcore-kctl`)       |
| `GetVm`                | kctl (`kcore-kctl`)       |
| `ListVms`              | kctl (`kcore-kctl`)       |
| `ListNodes`            | kctl (`kcore-kctl`)       |
| `GetNode`              | kctl (`kcore-kctl`)       |
| `ApplyNixConfig` (admin) | kctl (`kcore-kctl`)     |

### Node-agent RPC authorization

| RPC Method                | Allowed Callers                        |
|----------------------------|----------------------------------------|
| `ApplyNixConfig`           | Controller (`kcore-controller`)        |
| `InstallToDisk`            | kctl (`kcore-kctl`)                    |
| `ListDisks`                | kctl or Controller                     |
| `ListNetworkInterfaces`    | kctl or Controller                     |
| `GetVm`                    | kctl or Controller                     |
| `ListVms`                  | kctl or Controller                     |
| `GetNodeInfo`              | kctl or Controller                     |

When TLS is disabled (`--allow-insecure`), authorization checks are skipped since no certificates are available to inspect.

## Input Validation

### Nix config injection prevention

`nixgen::generate_node_config` escapes all user-controlled strings interpolated into Nix expressions:

- `nix_escape()` handles `\` -> `\\`, `"` -> `\"`, `${` -> `\${` (prevents Nix string interpolation)
- `sanitize_nix_attr_key()` strips attribute keys to `[a-zA-Z0-9_-]` only

This prevents an attacker with control over VM names, image paths, or network values from injecting arbitrary Nix code that would execute as root during `nixos-rebuild`.

### Disk path validation

All disk paths (`os_disk`, `data_disks`) in `InstallToDisk` are validated to:

- Start with `/dev/`
- Contain no `..` path traversal
- Contain no whitespace

This prevents command injection through the `install-to-disk` subprocess.

## Async Safety

Blocking I/O (filesystem reads/writes, subprocess execution, `/proc` reads) is offloaded from the tokio async runtime via `tokio::task::spawn_blocking` to avoid starving async tasks.

The SQLite database uses `std::sync::Mutex<Connection>` with a safe `lock_conn()` helper that returns a proper error on mutex poisoning instead of panicking the server.

## Known Limitations

- **No certificate rotation workflow** -- certs must be manually regenerated and redistributed before expiry
- **No CRL/OCSP** -- revoked certificates are not checked
- **No rate limiting** on gRPC endpoints
- **SQLite is single-writer** -- acceptable for the expected scale (tens of nodes), but limits concurrent write throughput

## Developer Workflow

### Entering the development shell

The project uses a Nix flake for reproducible builds. All Rust toolchain commands must be run inside the development shell:

```bash
nix develop
```

This provides `cargo`, `rustc`, `clippy`, `rustfmt`, and all native dependencies (e.g., SQLite, protobuf compiler).

### Running checks

From within `nix develop`:

```bash
# Build all crates
cargo build --workspace

# Run all tests (39 tests across 3 crates)
cargo test --workspace

# Lint with clippy
cargo clippy --workspace

# Audit dependencies for known vulnerabilities
cargo audit

# Format check
cargo fmt --check

# Run a specific crate's tests
cargo test -p kcore-controller
cargo test -p kcore-node-agent
cargo test -p kcore-kctl

# Run tests matching a pattern
cargo test -p kcore-controller -- nixgen
cargo test -p kcore-node-agent -- auth
```

If `cargo` is not on your `PATH`, you can run commands directly through Nix:

```bash
nix develop --command cargo test --workspace
nix develop --command cargo clippy --workspace
```

### Security-relevant test coverage

Key test areas to be aware of:

- **`nixgen::tests`** -- verifies Nix string escaping and attribute key sanitization, including injection attempts
- **`auth::tests`** -- verifies CN matching logic (exact match, prefix match, rejection of spoofed CNs)
- **`admin::tests`** -- verifies disk path validation, PKI file permissions (`0o600`/`0o644`), bootstrap write behavior
- **`pki::tests`** -- verifies cluster PKI generation, conditional controller key inclusion, mTLS handshake acceptance/rejection
- **`client::tests`** -- end-to-end mTLS connect tests (valid cert succeeds, untrusted cert fails)
